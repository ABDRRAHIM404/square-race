# Maze Runners — Implementation Plan

> Companion to `BUILD_SPEC.md`. Read `CLAUDE.md` first, then `BUILD_SPEC.md` for full detail on every
> mechanic. This document is the **execution order** — follow phases strictly in sequence. Do not start a
> phase until the previous phase's checkpoint passes. Do not jump ahead to polish or editor work while a
> core-engine phase is still failing its checkpoint.

Each phase lists: what to build, what NOT to build yet, and the checkpoint that must pass before moving on.
If a checkpoint fails, stop and fix it — do not carry a broken layer into the next phase.

---

## Phase 0 — Project Scaffold

**Build:**
- Folder structure from `BUILD_SPEC.md` Section 10.
- `package.json`, `capacitor.config.json`, `.gitignore` (with `android/` ignored). Use current Capacitor 8.x
  (`@capacitor/core`, `@capacitor/android`, `@capacitor/cli`) — do not pin to an old major version.
- `.github/workflows/build-apk.yml` (carry over from original project's CI approach: `npx cap add android`,
  `npx cap sync android`, `gradlew assembleDebug`, upload artifact). CI runner: Node.js 22+ (LTS), JDK 17.
- Confirm `android/variables.gradle` (generated by `npx cap add android`) lands on `minSdkVersion 24`,
  `compileSdkVersion`/`targetSdkVersion 36` per the current Capacitor 8.x scaffold — per `BUILD_SPEC.md`
  Section 9. Do not manually downgrade these.
- Empty `www/index.html` + `www/style.css` with full-screen, no-scroll, portrait mobile layout and a canvas
  element, using CSS safe-area-inset variables for edge-to-edge handling from the start (Section 9). No game
  logic yet.

**Do NOT build yet:** any maze data, physics, rendering, UI screens.

**Checkpoint:** Repo pushed to GitHub, Actions workflow runs and produces a debug APK artifact (even though
the app is just a blank canvas). This proves the pipeline works before any game complexity is added.

## Phase 1 — Maze Data Format + Default Maze

**Build:**
- `engine/mazeFormat.js`: the maze data shape from `BUILD_SPEC.md` Section 3, plus a validation function
  (checks the object has all required fields in the right shape — this is separate from *path* validation,
  which comes in Phase 6).
- `mazes/default-maze.js`: one hand-authored maze using this format — simple winding corridor, one dead end,
  clear start/exit. This ships as the app's only built-in maze.
- `engine/renderer.js`: takes a maze data object, draws walls/start/exit/dead-ends to canvas. Static only —
  no squares, no animation yet.

**Do NOT build yet:** physics, loot, bricks, pursuing wall, any UI screens beyond a bare page that renders
the default maze on load.

**Checkpoint:** Loading the app renders the default maze correctly on a static canvas, matching the
snap-to-grid winding-corridor look described in the spec.

---

## Phase 2 — Core Physics: Movement + Collision Only

**Build:**
- `engine/physics.js`: 4 colored squares spawn at the start point, move at their speed tiers, reflect off
  outer boundary and corridor walls (billiard-style), frame-rate independent via `deltaTime`.
- Minimal `engine/race.js`: detects when a square reaches the exit, records finish order. No elimination,
  no loot, no bricks, no pursuing wall yet.
- Minimal race loop wired to the Renderer from Phase 1 so this is visually watchable (not just console logs).

**Do NOT build yet:** knife/shield, bricks, pursuing wall, dead-end kill logic, any screens beyond the raw
race loop.

**Checkpoint:** Run the default maze. All 4 squares bounce correctly through the corridor using only
straight-line billiard physics, with correct reflection off every wall, and all 4 eventually reach the exit
in some order with placements recorded. No square passes through a wall. No square gets permanently stuck
against a wall.

## Phase 3 — Mechanics: Loot, Bricks, Pursuing Wall

Build these three independently and verify each in isolation before combining, ideally on small
purpose-built test mazes (in addition to the default maze) that make each mechanic easy to observe.

**3a. Loot — Knife & Shield**
- Spawn zones near-but-not-adjacent to exit (per Section 4.2).
- Knife: reusable on pickup, eliminates any square touched thereafter.
- Shield: absorbs exactly one knife hit, then breaks.
- Checkpoint: on a test maze, confirm knife carrier eliminates multiple squares across multiple touches
  (not consumed after first use), and a shielded square survives exactly one knife hit then becomes
  vulnerable again.

**3b. Breakable Bricks**
- Brick obstacles behave as solid walls until touched, then break/disappear permanently for the rest of
  that race.
- Checkpoint: on a test maze, confirm a brick blocks/reflects squares before being touched, and permanently
  opens the path after being touched.

**3c. Pursuing Wall**
- Follows behind the pack at constant 20% of square speed.
- Travels **along the corridor path itself** (advancing through the same corridor geometry from the maze
  data, following every turn/branch/dead end in sequence) — not a straight-line sweep across the canvas.
- **Fills the full width of the corridor floor** as it advances — once its leading edge has passed a point,
  that entire corridor cross-section is sealed, not just a thin line.
- Default behavior = acts exactly like a normal wall (squares bounce off the leading edge, no elimination)
  everywhere in the still-open corridor except:
- Dead-end kill: only eliminates a square when that square is in a dead end (per maze data's dead-end
  markers) and the wall's advancing front has sealed off the only way out, with no path left for the square.
- Checkpoint: on a test maze with an open corridor and a separate dead-end branch — confirm (1) the wall's
  leading edge visibly tracks the corridor's turns rather than moving in a straight line, (2) the filled
  region behind the wall spans the full corridor width with no gaps, (3) a square touching the wall in the
  still-open corridor just bounces off unharmed, (4) a square that ends up trapped in the dead-end branch is
  eliminated once the wall seals it in, and (5) a square that exits a dead-end branch before the wall seals
  it is NOT eliminated.

**3d. Combine**
- Run all three mechanics together on the default maze plus test mazes.
- Checkpoint: no mechanic interferes with another (e.g. a broken brick doesn't block loot pickup, a
  knife-carrier can still be caught by the pursuing wall in a dead end, etc.). Elimination placements
  (Section 4.6: eliminated squares placed last) are recorded correctly.

**Do NOT build yet:** speed-boost item (Phase 8 polish), any UI screens beyond the raw race loop, editor.

## Phase 4 — Race UI Screens + Scoring

**Build:**
- Title, Select, Stage Intro, Race (with HUD + mute button, safe-zone layout deferred to Phase 8 but leave
  room for it), Stage Result (with Quick-Restart), Final Result screens per `BUILD_SPEC.md` Section 6.
- Scoring: 4/3/2/1 points by placement, cumulative podium at Final Result.
- Flexible session length: for now, a session = however many mazes are passed into it (hardcode "just the
  default maze, once" as the only available flow until Phase 5 storage exists — do not hardcode "5 stages"
  anywhere).
- `audio.js`: Web Audio sfx for key events (collision, elimination, finish, UI taps), mute toggle wired to
  the Race screen's mute button.

**Do NOT build yet:** Maze Library screen (needs storage from Phase 5), editor, slow-mo/freeze-frame,
speed-boost.

**Checkpoint:** Full playable loop start to finish using only the default maze: Title → Select → Stage Intro
→ Race → Stage Result → Final Result, with correct scoring, working mute, and working Quick-Restart.

## Phase 5 — Storage: `localStorage` + Export/Import

**Build:**
- `storage/mazeStorage.js`: save/load/list/delete/duplicate against `localStorage` (or Capacitor's storage
  API), keyed per maze, storing the Section 3 data format.
- Export: per-maze and export-all, to JSON file/text/native share sheet.
- Import: parse JSON (file or pasted text), validate against `mazeFormat.js` from Phase 1, write to storage.
- Maze Library screen: lists saved mazes (plain list is fine for now — thumbnails come in Phase 7), lets the
  player pick which maze(s) go into a session.

**Do NOT build yet:** editor itself (this phase only needs storage + a way to select mazes for a session).

**Checkpoint:** Save the default maze under a new name via a manual test call, close/reopen the app, confirm
it's still listed. Export it, clear storage, import it back, confirm it round-trips identically. A session
can now be started from any combination of saved mazes chosen in the Library screen.

---

## Phase 6 — Maze Editor Core

**Build, in this order:**
1. Editor canvas using the same `renderer.js` from Phase 1 for live preview — grid-based wall drawing,
   snap-to-grid.
2. Set start point, exit point.
3. Mark dead ends.
4. Place bricks, place loot spawn zones (enforce near-but-not-adjacent-to-exit constraint).
5. Path validation: confirm a valid start-to-exit path exists before allowing save; block/warn otherwise.
6. Save flow into `mazeStorage.js` from Phase 5 (name the maze, write to storage).

**Do NOT build yet:** test-play button, thumbnails, duplicate/clone, time estimator, speed-boost toggle —
these are Phase 7.

**Checkpoint:** Design a brand-new maze entirely through the editor UI (not hand-written data), save it, then
play it through the full race flow from Phase 4 with zero changes to any engine/renderer/race-UI code. This
is the single most important checkpoint in the whole plan — it proves the architecture requirement from
`BUILD_SPEC.md` Section 2.

---

## Phase 7 — Editor Extras

**Build:**
- Test-play button: runs the in-progress maze directly through the real race engine from within the editor.
- Auto-generated thumbnail: small preview render, stored/shown in the Maze Library from Phase 5.
- Duplicate & tweak: clone an existing maze as a new editable copy.
- Built-in time estimator: rough completion-time estimate shown while editing, targeting 60–90s.
- Speed-boost item toggle: per-maze on/off; when on, adds the third loot type (temporary/one-time speed
  increase) to that maze's loot spawn logic.

**Checkpoint:** From the editor: test-play a maze without leaving the editor; confirm thumbnails appear
correctly in the Library; clone a maze and confirm the clone is independently editable/saveable; confirm the
time estimate is in the right ballpark against an actual test-play; confirm toggling speed-boost on adds a
working pickup with a visible speed effect, and off removes it.

## Phase 8 — Recording Polish

**Build:**
- Slow-mo/freeze-frame (~0.3s hit-stop) on kill events and each square's finish moment.
- Portrait 9:16 safe zones: audit all in-race HUD elements (score, mute button, etc.) and reposition so
  none sit in the very top/bottom margins that TikTok's own UI overlaps. Use CSS safe-area-inset variables
  (per `BUILD_SPEC.md` Section 9) so the same layout also correctly avoids the system status/navigation
  bars under Android's edge-to-edge display behavior, not just TikTok's overlay.

**Checkpoint:** Play a full race and visually confirm hit-stop triggers cleanly on both a kill and a finish
without breaking game state afterward (race continues/ends correctly). Confirm on a 9:16 portrait viewport
that no HUD element sits in the reserved top/bottom safe-zone margins, both in a browser test and in the
actual installed Android APK (where the system bars are real, not simulated).

---

## Phase 9 — Full Regression Pass

**Build:** nothing new — this is verification only.

**Checkpoint, run in order:**
1. Fresh install / cleared storage → app opens to Title with only the default maze available.
2. Design a new maze in the editor, save it, confirm it appears in the Library with a thumbnail.
3. Run a session using a mix of the default maze and the new maze; confirm scoring accumulates correctly
   across mazes and the Final Result podium is correct.
4. Confirm knife, shield, bricks, and pursuing-wall dead-end logic all still behave correctly (spot-check
   against Phase 3 checkpoints).
5. Confirm Quick-Restart, Export, Import, Duplicate all still work end to end.
6. Push to `main`, confirm GitHub Actions produces a working debug APK, install it on a device, and confirm
   the full flow above also works in the native Android build (not just the browser).

Only after Phase 9 passes completely is the rebuild considered done.
